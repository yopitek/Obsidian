## 7. Claude Code CLI 連線除錯

### 7.1 問題描述

Claude Code CLI 連線本地 Qwen LLM 非常緩慢，出現 "invalid api key" 錯誤。

### 7.2 根因分析

| # | 問題 | 嚴重性 |
|---|------|--------|
| 1 | `~/.openclaw-userb/openclaw.json` 指向舊的 port 8000 | ⭐⭐⭐ 高 |
| 2 | Qwen 模型響應在 `reasoning_content` 而非 `content` | ⭐⭐⭐ 高 |
| 3 | 終端環境變數被舊的 shell 會話繼承 | ⭐⭐ 中 |
| 4 | 多個配置文件包含舊的設定 | ⭐⭐ 中 |

### 7.3 除錯步驟

**Step 1: 驗證 llama.cpp 服務器運行**
```bash
ps aux | grep llama-server
ss -tlnp | grep 8081
```

**Step 2: 直接測試 API**
```bash
curl -s http://localhost:8081/v1/models \
  -H "Authorization: Bearer local-llama-key" | jq '.data[].id'
```

**Step 3: 驗證環境變數**
```bash
env | grep -iE "anthropic"
```

**Step 4: 修復 openclaw.json**
```bash
# 舊的設定 (錯誤):
"baseUrl": "http://localhost:8000/v1"

# 新的設定 (正確):
"baseUrl": "http://localhost:8081/v1"
```

**Step 5: 停用 Qwen thinking**
- 在 `~/run-llama-server.sh` 加入 `--reasoning off`
- 測試結果：`content` 有内容，`reasoning_content` 為 null

### 7.4 修復的檔案

| 檔案 | 變更 |
|------|------|
| `~/.openclaw-userb/openclaw.json` | baseUrl 從 8000 改為 8081 |
| `~/run-llama-server.sh` | 新增 `--reasoning off` |
| `~/.bashrc` | ANTHROPIC* 修正至 8081 |
| `~/.zshrc` | OPENAI* 修正至 8081 |

3. **Shell 繼承**：舊的終端會話會繼承錯誤的環境變數，需要在新的會話中設置正確的變數

## 12. Mac Mini LLM 伺服器設定

### 12.1 概述

**目的：** 使用 Mac Mini 作為另一台 LLM 伺服器，透過 Tailscale VPN 提供分散式推理服務。

**Tailscale IP：** `100.1.114.26`（注意：原輸入有誤，正確格式為 `100.1.114.26`）

### 12.2 推薦方案

#### 方案 A：macOS native llama.cpp（推薦）

**優勢：**
- 利用 Mac Mini 的統一記憶體（Unified Memory）
- Metal GPU 加速（Apple Silicon M-series）
- 輕量級部署，無需 Docker

**設定步驟：**

1. **安裝 Homebrew**（如未安裝）：
```bash
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"
```

2. **安裝 llama.cpp**：
```bash
brew install llama.cpp
```

3. **下載模型**（Qwen3.5-35B 或其他）：
```bash
# 建立模型目錄
mkdir -p ~/models

# 使用 huggingface-cli 或直接下載
huggingface-cli download unsloth/Qwen3.5-35B-A3B-GGUF Qwen3.5-35B-A3B-UD-Q4_K_XL.gguf --local-dir ~/models/
```

4. **啟動伺服器**：
```bash
llama-server -m ~/models/Qwen3.5-35B-A3B-UD-Q4_K_XL.gguf \
  --host 0.0.0.0 \
  --port 8081 \
  --api-key macmini-llm-key \
  -ngl 999 \
  --ctx-size 131072
```

5. **驗證 API**：
```bash
curl -s http://100.1.114.26:8081/v1/models \
  -H "Authorization: Bearer macmini-llm-key"
```

#### 方案 B：Docker 部署（跨平台一致性）

**優勢：**
- 環境一致性
- 易於複製部署
- 隔離性更好

**設定步驟：**

1. **安裝 Docker Desktop for Mac**

2. **建立 Dockerfile**：
```dockerfile
FROM ghcr.io/ggerganov/llama.cpp:server

COPY Qwen3.5-35B-A3B-UD-Q4_K_XL.gguf /models/

CMD ["-m", "/models/Qwen3.5-35B-A3B-UD-Q4_K_XL.gguf", \
     "--host", "0.0.0.0", \
     "--port", "8081", \
     "--api-key", "macmini-llm-key", \
     "-ngl", "999"]
```

3. **建置與啟動**：
```bash
docker build -t llama-macmini .
docker run -d \
  --name llama-server \
  -p 8081:8081 \
  -v ~/models:/models \
  llama-macmini
```

## 9. Claude Code CLI + LiteLLM Bug Fix 完整記錄

### 9.1 問題描述

Claude Code CLI 連線本地 Qwen LLM 失敗，出現以下錯誤：
- "invalid api key"
- "There's an issue with the selected model"
- 登入提示（要求選擇登入方式）

### 9.2 根本原因分析

經過數小時除錯，發現以下根本原因：

| # | 問題 | 原因 |
|---|------|------|
| 1 | API 端點格式不相容 | Claude Code CLI 需要 `/v1/messages` 端點，llama.cpp 只提供 `/v1/chat/completions` |
| 2 | 模型名稱驗證失敗 | Claude Code CLI 會驗證模型名稱是否是已知的 Anthropic 模型 |
| 3 | API Key 格式不符 | Claude Code CLI 預期 Anthropic API Key 格式 |
| 4 | KV Cache 失效 | 未設定 `CLAUDE_CODE_ATTRIBUTION_HEADER=0` 導致推論速度慢 90% |
| 5 | 舊配置覆蓋 | 多個配置文件包含舊的 vLLM 設定（port 8000） |

### 9.3 除錯過程（時間線）

```
13:00 - 發現 Claude Code CLI 顯示 "invalid api key"
13:15 - 檢查 ~/.bashrc、~/.zshrc，發現環境變數指向錯誤的 port 8000
13:30 - 修復環境變數指向 port 8081
13:45 - 測試 curl API 正常，但 Claude Code CLI 仍然失敗
14:00 - 發現 Qwen 模型回應在 reasoning_content 而非 content
14:15 - 加入 --reasoning off 參數
14:30 - API 回應正常，但 Claude Code CLI 仍顯示模型驗證失敗
14:45 - 嘗試各種模型名稱（Qwen3.5-35B-A3B、claude-sonnet-4-6 等）
15:00 - 查閱官方文檔，發現需要 LiteLLM 作為代理
15:30 - 安裝 LiteLLM 並配置代理
16:00 - 測試 LiteLLM /v1/messages 端點成功
16:15 - 配置 Claude Code CLI 連接 LiteLLM
16:30 - ✅ 成功！Claude Code CLI 可以使用本地 Qwen 模型
```

### 9.4 解決方案

使用 **LiteLLM** 作為代理層，將 llama.cpp 的 `/v1/chat/completions` 端點轉換為 Claude Code CLI 需要的 `/v1/messages` 端點。

#### 系統架構

```
Claude Code CLI (localhost:4000)
    ↓ HTTP /v1/messages
LiteLLM Proxy (port 4000)
    ↓ HTTP /v1/chat/completions
llama.cpp Server (port 8081)
    ↓ CUDA
Qwen3.5-35B-A3B Model (本地 GPU)
```

### 9.8 Lesson Learned（Bug Fix）

1. **Claude Code CLI 不支援直接連接 llama.cpp**
   - 需要 `/v1/messages` 端點格式
   - llama.cpp 只提供 `/v1/chat/completions`
   - **解決方案**：使用 LiteLLM 作為代理

2. **多個配置來源的問題**
   - ~/.bashrc、~/.zshrc、~/.openclaw-userb/ 都可能設定環境變數
   - 舊的終端會話會繼承錯誤的變數
   - **解決方案**：使用 wrapper script 強制設定正確變數

3. **Qwen 模型的 thinking 輸出**
   - 回應在 `reasoning_content` 而非 `content`
   - 使用 `--reasoning off` 參數停用

4. **KV Cache 失效問題**
   - Claude Code CLI 預設發送 Attribution Header
   - 導致每次請求都重新處理 KV Cache
   - **解決方案**：設定 `CLAUDE_CODE_ATTRIBUTION_HEADER=0`

5. **API Key 格式**
   - LiteLLM 接受任意 API Key
   - 使用 `sk-no-key-required` 作為虛擬值

6. **跳過登入**
   - 在 `~/.claude.json` 加入 `hasCompletedOnboarding: true`
   - 加入 `primaryApiKey: "sk-dummy-key"`

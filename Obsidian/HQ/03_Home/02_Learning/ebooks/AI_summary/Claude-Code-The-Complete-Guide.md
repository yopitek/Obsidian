# Claude Code：完整指南摘要

## 一句話總結

這本書的核心不是「用 AI 幫你寫程式」，而是「把 AI 當成一位能力很強、但需要被正確管理的新同事」，再透過規則、上下文、工具與流程，把它變成穩定可用的生產力系統。

## 重點摘要

1. Claude Code 的真正價值，不只是生成程式，而是能直接操作終端機、檔案、Git 與外部工具，完成完整工作流。
2. 最重要的概念是 `CLAUDE.md`：它像 AI 的工作地圖、團隊規範與長期記憶入口，能讓每次新 session 都快速進入狀態。
3. 高品質輸出不是靠一次神 prompt，而是靠持續累積上下文、修正錯誤、補規則，形成 context engineering。
4. Skills、Hooks、MCP 是三個關鍵延伸機制：Skills 教 AI 做事，Hooks 在重要節點做檢查，MCP 把 AI 連到外部世界。
5. 多 Agent 協作與專案化工作流，代表 AI 不只是聊天工具，而是可被編排的執行系統。
6. 書中也強調限制與陷阱：AI 很強，但仍會誤解上下文、做出錯誤假設、在複雜專案裡失控，所以流程與驗證不可少。

## 重要概念

### 1. `CLAUDE.md` 是 AI 的憲法

不是把所有資訊塞進去，而是把最常犯錯、最重要的規則、專案結構、常用指令放進去。好的 `CLAUDE.md` 是從真實錯誤中長出來的。

### 2. 把 AI 當成「聰明的新同事」

它能力高，但不熟悉你的脈絡。你要做的不是一直懷疑它，而是給它正確背景、清楚目標、可執行邊界。

### 3. Context engineering 比 prompt engineering 更重要

真正影響品質的，是整體上下文設計：檔案、規範、工具權限、專案結構、歷史錯誤，而不是一句漂亮指令。

### 4. Skills / Hooks / MCP 三件組

- **Skills**：封裝常見工作方法。
- **Hooks**：在提交、修改、執行前後加入保護機制。
- **MCP**：把 GitHub、資料庫、瀏覽器、第三方服務變成 AI 可直接操作的工具。

### 5. AI coding 的瓶頸常在流程，不在模型

如果沒有清楚結構、驗證、協作邏輯，再強的模型也會持續重犯同樣錯。

## 日常生活如何運用

1. 把常做的事寫成固定流程，例如每週回顧、文章校稿、資料整理，讓 AI 不只是回答問題，而是照 SOP 幫你做事。
2. 為自己的常用工作資料夾建立一份簡短規則檔，像是檔案命名、輸出格式、儲存位置，減少重複溝通。
3. 遇到 AI 做錯時，不要只重問一次；把錯誤整理成規則，讓下一次表現更穩。
4. 把 AI 當成助理，不是神諭。凡是重要決策、金錢、法律、隱私相關內容，都要保留人工判斷。

## 工作上如何運用

1. 在團隊專案建立共享的 `CLAUDE.md`，把專案結構、指令、測試方式、禁止事項寫清楚，降低每位成員與 AI 的溝通成本。
2. 把重複性工作模組化成 Skills，例如 code review、文件整理、發文流程、需求拆解。
3. 用 Hooks 把品質控制自動化，例如變更後先跑測試、提交前檢查格式、產出前檢查敏感資訊。
4. 用 MCP 串 GitHub、資料庫、Notion、Slack 等工具，讓 AI 從「建議者」升級成「可執行的工作節點」。
5. 多 Agent 協作適合大型任務：研究、撰寫、校對、產圖、發布可拆成不同角色平行進行。

## 我認為這本書最值得帶走的 3 件事

1. AI 的品質，取決於你怎麼設計它的工作環境。
2. `CLAUDE.md` 不是文件，而是把經驗變成系統的地方。
3. 真正成熟的 AI 使用方式，不是偶爾問問題，而是把它納入日常工作流。
